Era 1 · Pre-2000

The Single-Core Era

One CPU. One core. One task at a time. And that was fine — for a while.

The Server of the 1990s
  • One physical socket on the motherboard
  • One core inside that socket — single execution unit
  • RAM: typically 16 MB to 256 MB
  • Clock speed: ~66 MHz (486 era) to ~1 GHz by 2000
  • OS installed directly on the hard disk — boots bare metal
  • One server = one OS = one application
What "Single Core" Means
  • CPU executes one instruction stream at a time
  • Multiple processes "share" the CPU via fast time-slicing — an illusion of parallel
  • True simultaneous execution: impossible
  • Software designed around this — fine for batch payroll, overnight reports
The Workload Reality: Most 1990s enterprise software was batch-oriented — payroll, end-of-day reports, overnight processing. Interactive users were few. One core was enough.
The Prediction · 1965

Moore's Law

Gordon Moore's 1965 observation became the engine that drove 50 years of computing progress

The Prediction
  • Gordon Moore (Intel co-founder, 1965): transistor count on a chip doubles approximately every 18–24 months
  • More transistors = more logic = faster, smarter CPU
  • Held remarkably true for over 50 years
  • Note: Moore predicted transistor doubling, not directly clock speed — but the two were tightly linked during the clock-speed era
The Real-World Numbers
  • 1971 Intel 4004: 2,300 transistors · 0.74 MHz
  • 1993 Pentium: 3.1 million · 66–200 MHz
  • 2000 Pentium 4: 42 million · 1.5 GHz
  • 2012 Ivy Bridge: 1.4 billion · 3.5 GHz
  • 2023 Apple M2 Ultra: 134 billion transistors
  • A 58-million-fold increase in 52 years
The Free Lunch: For decades, developers didn't need to optimise code. Just wait 18 months, buy the new CPU — software ran faster automatically. This era ended around 2004 when clock speed hit its thermal limit.
The Bottleneck · ~2004

The Thermal Wall

Clock speeds couldn't keep rising — physics intervened, forcing a complete rethink of CPU design

The Clock Speed Race
  • CPU performance = clock speed × work per cycle
  • Industry focused on clock: 66 MHz → 1 GHz → 3.8 GHz
  • Intel Pentium 4 "Prescott" hit 3.8 GHz in 2004
  • Power consumption: ~130 Watts — the chip ran scorching hot
Why It Stopped — Physics
  • Power = C × V² × frequency — higher clock = more heat
  • At ~4 GHz the chip could not be cooled reliably in a server rack
  • Intel abandoned its "Terahertz Transistor" roadmap targeting 10 GHz
  • Clock speeds dropped from 3.8 GHz back to ~2.4 GHz when multi-core arrived
  • Modern per-core clock: 3–5 GHz — barely above the 2004 peak
The Pivot: If we can't go faster per core — add more cores. One fast engine → many parallel engines. This shift changed how software had to be written forever.
The Solution · 2005 onwards

Multi-Core & Multi-Socket

More cores at slightly lower clock — same chip area, massively more parallel throughput

The Multi-Core Leap
  • 2005: Intel/AMD — first dual-core desktop CPUs
  • 2007: Quad-core mainstream (Core 2 Quad)
  • 2010: 6–8 core server CPUs common
  • 2020: AMD EPYC 7742 — 64 cores per socket
  • 2023: AMD EPYC Genoa — 96 cores per socket
  • Apple M2 Ultra: 24-core CPU + 76-core GPU on one chip
Memory Grew 750,000×
  • 1990: 16–64 MB typical server RAM
  • 2000: 512 MB – 2 GB
  • 2008: 16–32 GB
  • 2015: 256 GB – 1 TB
  • 2024: AMD EPYC / Intel Xeon — up to 12 TB per server
  • Entire databases now fit in RAM → 1,000× faster than disk reads
But Old Apps Couldn't Use It: Single-threaded software runs on one core while 63 others sit idle. Adding cores doesn't speed up sequential code. Average enterprise server utilisation: 5–15% — before virtualisation.
The Solution · VMware 1998

Virtual Machines — The Big Idea

What if one physical server could pretend to be many? The hypervisor made this possible.

The Key Abstraction
  • Bare metal: OS talks directly to physical CPU, RAM, disk, network
  • With virtualisation: a Hypervisor sits between hardware and OS
  • Hypervisor presents virtual hardware to each Guest OS
  • Guest OS doesn't know it's on virtual hardware
  • Multiple Guest OSes run simultaneously, each fully isolated
  • Pioneer: VMware (1998) for x86; IBM mainframes had this since the 1960s
VM Superpowers
  • 📸 Snapshots: capture complete VM state — roll back in seconds
  • 📋 Templates: build one "golden" VM, clone 50 copies in minutes
  • 🚚 vMotion: move a running VM to another host with zero downtime
  • ⚡ Hot-add: add CPU cores, RAM, or storage while VM is live
  • Utilisation jumps from 5% → 60–80% after consolidation
Deployment Speed: Bare metal: order hardware → wait weeks → rack → cable → OS install → 2–6 weeks total. VM from template: clone → start → configure network → 5–15 minutes. This speed changed the economics of scaling permanently.
The Foundation

VMs Power The Cloud

AWS, Azure, and GCP took VM technology, automated it with APIs, and sold it to the world

AWS EC2 (2006) — The Breakthrough
  • Rent a virtual machine by the hour — any size, any quantity
  • Elastic: scale from 1 to 10,000 VMs — and back — in minutes
  • VM templates (AMIs) — deploy identical copies instantly
  • All the VM superpowers delivered via a web API
  • No hardware to own, manage, cool, or depreciate
What the Cloud Delivered
  • Pay per minute of use — scale to zero at night, stop paying
  • Handle peak load without pre-buying idle capacity
  • Spin up a test environment in minutes (was weeks via ticket queue)
  • Geographic scale: deploy to 30+ regions globally in hours
  • Azure Virtual Machines, Google Compute Engine — same model
Summary

The Full Story

Five ideas — from single transistors to cloud computing

① Moore's Law → Free Speed
  • Transistor count doubled every 18 months for 50 years. Clock speeds rose 66 MHz → 3.8 GHz. Apps got faster automatically — no code changes needed.
② Thermal Wall → Multi-Core
  • ~2004: clock speed stopped at 3.8 GHz due to heat. Industry pivoted: lower clock, many more cores. Old single-threaded apps gained nothing — had to be rewritten.
③ Memory → 750,000× Growth
  • RAM grew from 16 MB (1990) to 12 TB (2024). Entire databases fit in memory. But legacy apps wasted nearly all of it — only 5–15% utilisation.
④ VMs → Efficient Utilisation
  • Hypervisor abstracts hardware → run 10–20 VMs per server. Snapshots, templates, vMotion, hot-add transformed IT operations. Utilisation: 5% → 70%+.
⑤ VMs + Scale = The Cloud
  • AWS, Azure, GCP automated VM technology at massive scale, delivered it via web APIs, and sold it to the world on a pay-per-minute basis. The cloud's elastic scaling, instant provisioning, and pay-per-use billing are all direct consequences of virtualisation technology. Moore's Law + Virtualisation, delivered as a utility.
