Software Engineering · Evolution of Practice

From No Process
to Platform Engineering

Nine phases over seventy years — how the software industry learned, the hard way, that the way you build and deliver software is just as important as what you build. Every transition was forced by the failure modes of the previous era.

Waterfall → Agile CI / CD DevOps DevSecOps Platform Engineering
Overview

Nine Phases, One Direction

More automation · faster feedback · quality shifted earlier · ownership spread wider. Every phase directly addressed the failure mode that preceded it.

The First Problem: Chaos → Structure
  • Phases 1–3 solved the lack of discipline and repeatability
  • Waterfall gave process — but too rigid; change was impossible mid-flight
  • Iterative waterfall added feedback loops — but phases still ran months each
The Second Problem: Speed → Quality
  • Agile broke waterfall's long cycles into sprints — but deploy was still manual
  • CI automated testing on commit; CD automated deployment to production
  • Features could now ship in hours, not months
The Third Problem: Silos → Shared Ownership
  • DevOps merged development and operations — "you build it, you run it"
  • DevSecOps embedded security throughout — not a gate at the end
  • Platform Engineering gives self-serve infra — no ops ticket queue
Phase 1 · 1940s–1960s

No Process — Heroic Individual Effort

Computing's earliest era had no methodology, no version control, no testing discipline. Programming was a craft practised by a tiny elite, and every project relied on the skill and heroism of individuals.

How Work Was Done
  • Programs written directly in machine code or assembly — no high-level languages
  • Punch cards: each card = one instruction; drop the deck = start over
  • Batch submitted to operators overnight; errors returned next morning
  • No version control — the only copy was in the programmer's head and on the deck
  • No documentation standard; knowledge lived with individuals
  • Testing: manually check output against expected results on paper
Why It Broke Down
  • The Software Crisis (1968): NATO conference formally named the problem — projects were late, over budget, unreliable, and impossible to maintain
  • Software projects grew too large for individual heroism to sustain
  • No repeatability — you couldn't hire a new person and expect the same result
  • No way to estimate cost or time — every project was unique
  • IBM System/360 required coordinating hundreds of programmers — forced the invention of structured methodology
The Scale Problem: Fred Brooks documented this in The Mythical Man-Month (1975) — adding more programmers to a late project makes it later. Software complexity doesn't scale linearly. You need process, not just people.
Phase 2 · 1970s–1990s

Waterfall — Process Arrives, Rigidity Follows

Winston Royce (1970) described a sequential phase-based model. Industry adopted it as the answer to the software crisis — it brought discipline, but at the cost of all flexibility.

What Waterfall Gave Us
  • Requirements: document everything the customer wants — freeze and sign off
  • Design: architect the system, define interfaces, database schema
  • Implementation: code all features against the spec
  • Verification: test everything — QA team runs test scripts
  • Maintenance: fix bugs in production, handle change requests
  • Total cycle: typically 12–18 months from contract to go-live
The Fatal Flaw
  • Requirements are signed off at the start — but customers don't know what they want until they see it
  • Change mid-project requires re-phasing — expensive and politically painful
  • Bugs found in the test phase cost 100× more to fix than if caught in design
  • Customer sees no working software for 12+ months — then discovers it's wrong
  • Integration testing happens last — when all the complexity lands at once
Phase 3 · 1980s

Iterative Waterfall — Feedback Loops Added

Barry Boehm's Spiral Model (1986) and other iterative approaches accepted that phases needed feedback connections. Each iteration reduced risk — but the core phase-gated structure remained.

What Changed from Waterfall
  • Phases now explicitly allowed feedback to previous phases
  • Spiral Model: each loop through the spiral adds scope and reduces risk
  • Prototyping introduced — show something to the customer earlier
  • Risk-driven rather than document-driven — highest-risk items tackled first
  • Incremental delivery: ship a partial system, learn, iterate
  • Rational Unified Process (RUP): 4 phases × multiple iterations each
Still Broken
  • Iterations still measured in months — not days or weeks
  • Each iteration still had its own mini-waterfall phases
  • Heavy up-front documentation and process overhead
  • Teams still siloed: separate requirements, design, dev, test groups
  • Deploy still manual and rare — a separate "release management" function
  • Feedback loops on paper — in practice, going back was still expensive
The Core Problem Remained: All iterative waterfall variants kept the fundamental assumption that you could plan in detail before building. The Agile Manifesto (2001) rejected this assumption entirely — "responding to change over following a plan."
Phase 4 · Agile Manifesto · February 2001

The Agile Manifesto — A Philosophy, Not a Process

Seventeen practitioners at Snowbird, Utah produced four values and twelve principles. Not a framework — a set of beliefs about how good software development actually works.

The Four Values
  • Individuals and interactions over processes and tools
  • Working software over comprehensive documentation
  • Customer collaboration over contract negotiation
  • Responding to change over following a plan
  • The items on the right still have value — but we value the items on the left more.
Key Principles (of twelve)
  • Deliver working software frequently — weeks, not months
  • Welcome changing requirements, even late in development
  • Business people and developers work together daily
  • Build projects around motivated individuals; give them the environment and trust
  • The most efficient way to convey information: face-to-face conversation
  • Working software is the primary measure of progress
  • Continuous attention to technical excellence and good design enhances agility
  • At regular intervals, reflect on how to become more effective
What Made Agile Different: It acknowledged that requirements are discovered, not specified. You cannot know everything upfront. The Manifesto legalised change and made customer collaboration a first-class value rather than a contractual inconvenience.
Phase 4 · Agile Frameworks

Agile in Practice — Scrum, Kanban, XP

The Manifesto inspired several concrete frameworks. Each operationalised the four values differently — short iterations, continuous flow, or technical rigour.

Scrum (Most Common)
  • Sprint: 1–2 week timebox — plan, build, test, demo, retrospect
  • Roles: Product Owner (what), Scrum Master (how), Dev Team (build)
  • Artefacts: Product Backlog, Sprint Backlog, Working Increment
  • Definition of Done: tested, reviewed, merged, demo-able
Kanban & XP
  • Kanban: continuous flow, WIP limits, no fixed sprint — visualise work in progress
  • XP (Extreme Programming): pair programming, test-first (TDD), continuous integration, small releases, collective code ownership
  • XP was the technical answer to Agile — Scrum handled process, XP handled engineering practices
Phase 5 · 2005 onwards

Continuous Integration

Martin Fowler formalised CI in 2000: every developer integrates their work into a shared mainline at least daily, and every integration is automatically verified by an automated build and test suite. Before CI, integration was a project phase — a painful multi-day event.

The CI Contract
  • Every git push triggers the pipeline automatically — no manual action
  • Build: compile, resolve dependencies, produce a testable artifact
  • Unit tests: fast (seconds), test one function in isolation
  • Integration tests: test multiple components working together
  • Quality gate: build fails → no merge → developer is notified immediately
  • Build must pass in under 10 minutes to be useful as fast feedback
The Test Pyramid
  • 🏔 Top — E2E tests (few): slow, brittle, expensive. Test the full journey. Catch regressions. Run on staging.
  • 🔷 Middle — Integration tests: test service boundaries, database queries, message queues. Minutes to run.
  • 🟩 Base — Unit tests (many): fast (ms each), test one function. Run on every commit. 70%+ of all tests.
  • Don't invert the pyramid — slow E2E suites kill CI feedback speed
What CI Changed: "Integration hell" — the painful multi-day merge at the end of a sprint — disappeared. Bugs are caught within minutes of being introduced, when the context is still fresh. The developer who broke the build fixes it immediately, not weeks later.
Phase 6 · 2009 onwards

Continuous Delivery & Deployment

Jez Humble and Dave Farley (2010): every change that passes all tests is releasable to production at any time. CD takes CI's artifact and automates the entire path to production — safely.

Feature Flags

Release ≠ Deploy. Code ships to production but stays hidden behind a flag. Enable for 1% → 10% → 100% of users. Kill it instantly if metrics degrade.

Canary Releases

Route 1% of real traffic to the new version. Compare error rates, latency, and business metrics against the stable version. Auto-rollback on threshold breach.

Blue / Green Deploy

Two identical production environments. Switch traffic from Blue (old) to Green (new) in one instant. Blue stays live — rollback is just switching back. Zero-downtime guaranteed.

A/B Testing

Two versions of a feature simultaneously — 50% of users each. Measure which version achieves better business outcomes. Data-driven decisions, not opinion-driven.

Phase 7 · 2009 onwards · The Wall of Confusion

DevOps — Tearing Down the Wall

CI/CD automated the pipeline — but development and operations remained separate teams with opposing goals. DevOps addressed the cultural and organisational problem that automation alone couldn't fix.

Phase 7 · DevOps Practices

DevOps — What Actually Changed

DevOps is not a tool or a job title. It is an organisational culture shift with specific, measurable engineering practices — including testing disciplines that were systematically absent before.

Cultural & Organisational Changes
  • "You build it, you run it" — Werner Vogels, Amazon CTO (2006). Developers share on-call. They feel the pain of their own reliability failures.
  • Operations involved in sprint planning — infra requirements treated as user stories
  • Developer access to production metrics — understands real-world performance, not just test environments
  • Blameless post-mortems: when something breaks, fix the system, not the person
  • Infrastructure as Code (IaC) — ops changes go through the same Git review process as code
  • DORA metrics: Deployment Frequency, Lead Time, MTTR, Change Failure Rate
Performance & Load Testing — Finally in the Pipeline
  • Before DevOps: performance testing was a separate engagement, run by a specialist team, once before a major release — too late to change the architecture
  • In DevOps pipelines: performance tests run on every build against a production-like environment
  • Load testing (k6, Gatling, JMeter): simulate thousands of virtual users — catch regression before it reaches production
  • Stress testing: find the breaking point — so you know your limits before users do
  • Soak/endurance testing: run at normal load for hours — catch memory leaks and connection pool exhaustion
  • Chaos engineering (Netflix Chaos Monkey): deliberately kill random services in production — prove the system handles failure gracefully
Operations Feedback Loop: Monitoring data — slow endpoints, high error rates, memory pressure — feeds directly into the next sprint's backlog. Ops improvements are not "tech debt tickets that never get done" — they are first-class work items prioritised alongside features.
Phase 8 · 2015 onwards

DevSecOps — Security Shifts Left

In DevOps, security remained a gate at the end — a "penetration test" performed by a separate team before release. DevSecOps embeds security at every stage, making it a shared responsibility of the whole team, not an audit function.

SAST

Static Application Security Testing — scans source code for vulnerabilities without running it. SQL injection, XSS, hard-coded secrets. Runs on every commit.

SCA

Software Composition Analysis — scans third-party dependencies for known CVEs (Common Vulnerabilities and Exposures). Your code may be clean; your libraries may not be.

Container Image Scan

Scans Docker images for OS-level CVEs in the base image and installed packages — before push to registry. Tools: Trivy, Grype, Snyk Container. Blocks deploy if critical CVE found.

DAST & Secrets Detection

DAST — Dynamic scanning of the running application for auth flaws, injection, misconfigurations. Secrets Detection — git-leaks, TruffleHog scan commit history for API keys, passwords committed by accident.

Phase 9 · 2020 onwards

Platform Engineering — Self-Serve Everything

At scale — 50+ microservice teams, hundreds of engineers — DevOps practices create a new bottleneck: every team reinvents the same CI/CD pipelines, Kubernetes configs, monitoring dashboards, and security policies. Platform Engineering solves this with an Internal Developer Platform (IDP).

The Problem at Scale
  • 50 teams × 3 pipelines each = 150 pipeline configs to maintain
  • Each team applies security policies differently — inconsistent risk posture
  • New developer: weeks to get a working local + cloud environment
  • "Cognitive load" of infrastructure kills feature delivery speed
The IDP Solution
  • Golden paths: opinionated, pre-approved templates — deploy a new service in 5 minutes
  • Service catalogue: every microservice documented, with runbooks, owners, SLOs
  • Self-serve environments: dev requests a staging env — IDP provisions it, policy-compliant, billed to their cost centre
  • GitOps: all deployments via ArgoCD/Flux — Git is the single source of truth
Platform Team as Product Team
  • The Platform Team treats developers as their customers
  • User research: what are the pain points? Where do developers lose time?
  • SLOs for the platform itself — uptime, provisioning latency, deployment success rate
  • Not a "gatekeeper" — a force multiplier. Less time on infra → more time on business features

Measuring Progress

DORA Metrics — Are You Actually Getting Better?

Google's DevOps Research and Assessment team identified four metrics that distinguish Elite performers from Low performers. These are the objective measures of how far along the journey a team has come.

Freq
Deployment Frequency
Elite: multiple times per day. Low: once per 6 months. The single best proxy for team maturity.
Hours
Lead Time for Changes
Elite: <1 hour from commit to production. Low: 1–6 months. Shorter = faster value delivery.
<15%
Change Failure Rate
Elite: 0–15% of changes cause incidents. Low: 46–60%. High frequency + low failure = the target.
<1hr
Mean Time to Recovery
Elite: recover from incidents in under an hour. Low: 1 week to 1 month. Drives blameless culture + chaos engineering.
The Counterintuitive Insight
  • High deployment frequency and low change failure rate are not in tension — elite teams achieve both simultaneously
  • Deploying more frequently actually reduces risk — smaller changes, easier to diagnose failures, faster rollback
  • Teams that deploy rarely have the highest change failure rates — each deployment is a massive, untested blast radius
  • The path to high reliability is more deploys, not fewer
Elite vs Low Performers (2023 DORA Report)
  • Deployment Frequency: Elite = multiple/day · Low = <1/month
  • Lead Time: Elite = <1 hour · Low = 1–6 months
  • Change Failure Rate: Elite = 0–15% · Low = 46–60%
  • MTTR: Elite = <1 hour · Low = 1 week–1 month
  • Elite performers are 127× more likely to meet or exceed organisational goals
Summary

Seven Key Ideas

If you carry nothing else from this presentation — carry these.

① Every Phase Was Forced

Nobody adopted Agile or DevOps for fashion. Each transition happened because the previous model demonstrably failed at scale. Understanding the failure mode explains the solution.

② Quality Shifts Left

Finding a bug in production costs 100× more than finding it in code review. Every evolution — CI, CD, DevSecOps — moves quality checks earlier. Left-shift is both a cost and a reliability strategy.

③ Deploy More to Fail Less

Counterintuitive but proven: smaller, more frequent releases carry lower risk. Each deployment is smaller, easier to diagnose, easier to rollback. Elite teams deploy 100s of times per day.

④ Silos Kill Quality

Waterfall silos created blame culture. Dev/Ops separation created "works on my machine." Security-at-the-end created late, expensive fixes. Every evolution broke down a silo.

⑤ Automation Is Not Optional

Manual testing, manual deployment, manual security review: each is a bottleneck that prevents the frequency needed for elite performance. Automate the pipeline; focus humans on judgement.

⑥ Culture Comes First — Tools Come Second · ⑦ Measure Outcomes, Not Activity

You cannot buy DevOps. Installing Jenkins without changing the Dev/Ops relationship installs a faster version of the same broken process. Culture — shared ownership, blameless retrospectives, customer focus — precedes the tooling. And always measure DORA metrics: deployment frequency, lead time, change failure rate, MTTR. Story points delivered are activity; working software in production is outcome.
