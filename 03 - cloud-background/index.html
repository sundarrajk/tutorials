Application Architecture — Before Cloud

Two Dominant Patterns

Every enterprise ran one or both of these architectural patterns. Understanding them is essential context for what we had to rethink when moving to the cloud.

3-Tier Web Architecture

Web Server — handles HTTP, serves static content.
Application Server — executes business logic.
Database Server — persistence and queries.

Each tier ran on dedicated physical or virtual machines with well-known, static IPs. Scaling meant adding more machines to a tier.

ESB / SOA Architecture

The Enterprise Service Bus (ESB) acted as a central message broker connecting disparate systems. Under Service-Oriented Architecture (SOA), each application exposed services consumed by others via the bus.

The ESB handled routing, transformation, and orchestration — avoiding brittle point-to-point wiring between every pair of applications.

On-Premises Network Layout

3-Tier Architecture in Practice

Every tier had fixed, assigned IP addresses. Other applications and services simply hard-coded or configured these addresses — they almost never changed.

Enterprise Integration Pattern

ESB / SOA — All Roads Lead to the Bus

Rather than applications wiring directly to each other (creating a spaghetti of point-to-point connections), the ESB centralised all inter-application communication through one well-known address and port.

Networking Fundamentals

TCP/IP — The Language of Applications

Applications communicated using Transmission Control Protocol / Internet Protocol (TCP/IP). Every server had a fixed IP address. Each application listened on a port number. Together, they formed the endpoint address:

192.168.1.10 : 8080
IP Address → which machine  ·  Port → which application

Any client that knew this pair could establish a connection. The channel was full-duplex — data could flow in both directions simultaneously, making it suitable for request/response, streaming, and long-lived connections alike.

Key TCP Properties

  • ✓Reliable delivery — TCP guarantees all packets arrive and are delivered in order
  • ✓Connection-oriented — 3-way handshake before any data is exchanged
  • ✓Full-duplex — both sides can send and receive simultaneously
  • ✓Static addressing — in on-premises setups, IPs rarely if ever changed
Deployment Lifecycle

Deploying an Application

There were two scenarios. Both required careful planning and execution, but the timeline differed dramatically depending on whether spare capacity already existed.

⏱ Key point: Even Option A could consume hours of skilled engineering time. Option B — which required procuring and setting up a physical server first — routinely stretched to days or weeks before the application was running in production.
Scaling & Migration

Adding an Instance or Moving an App

Even a routine scale-out (adding one more application-server instance) was effectively equivalent to a full new deployment. Moving an application from one server to another carried the additional risk of silently breaking every other application that had been configured with its old IP and port.

📅
Weekend windows only

Migrations ran Friday night through Sunday. Teams arrived Monday morning uncertain whether all systems would work as expected.

⏳
Hours to days of downtime

Application downtime during migrations regularly ran into hours. Complex estates with many dependencies could require 24–48 hours of downtime.

📋
Meticulous planning required

Every downstream dependency had to be mapped. Every configuration had to be updated. A single missed endpoint could cascade into a Monday morning crisis.

📈
Memory & CPU scaling too

Increasing RAM or CPU on a bare-metal server was a physical hardware task. Even on VMs it typically required a scheduled maintenance window and downtime.

Virtualisation Era

VMs Eased — But Did Not Solve — the Problem

With Virtual Machines, the OS and application-server stack was pre-baked into a reusable image. Deployment could begin from application install rather than bare-metal OS installation, compressing the cycle significantly.

✅ CPU and memory could now be increased by reconfiguring the hypervisor — no physical hardware work needed, and often no downtime.

Network and load-balancer configuration still needed updating with the new VM's IP and port. But overall the cycle shrank from weeks to days.

VM Deployment Steps

  1. 1Clone or spin up a pre-built OS/runtime VM image
  2. 2Deploy the application binary to the new VM
  3. 3Configure the application (data sources, queues, etc.)
  4. 4Update load balancer or web server with the new IP:port
  5. 5Start the server and validate all connections
The Core Bottleneck

The Real Roadblock: Hardware Procurement

💡 Key insight: Nobody wanted idle servers sitting unused — every machine was put to work. The moment you needed additional capacity, you had to buy new hardware, and that process took months from approval to production.
💰
Budget approvals

Internal finance sign-offs traversed multiple management layers. Unplanned capacity requests were especially slow.

🤝
Vendor negotiations

Price negotiation, lead-time commitments, and SLA agreements with hardware vendors consumed additional weeks.

🚚
Shipping & racking

Hardware shipped from manufacturer to data centre, physically racked, and cabled into the network switch fabric.

🔧
OS install & hardening

Only after all of the above could the OS installation and security hardening begin — before the application was even touched.

Enter the Cloud

The Solution: On-Demand "Infinite" Infrastructure

Cloud providers solved the hardware procurement problem entirely. With pre-baked VM images, self-service APIs, and extensive automation, standing up a new server collapsed from months to hours.

♾️
Effectively unlimited capacity

Need ten more servers? Spin them up in minutes. Need a hundred? Same API call. No procurement cycle, no waiting.

⚡
Hours, not months

CLI tools, APIs, and Infrastructure-as-Code (IaC) templates provision fully configured VMs in minutes. Deployment pipelines handle the rest automatically.

💳
Pay-as-you-go

Switch from CapEx (buy before you need it) to OpEx (pay for what you use). Scale down just as easily as you scale up.

Obstacles

Why Moving to Cloud Was Not Simple

The cloud promised fast, cheap capacity — but most enterprise application estates were not built to exploit it. Six common obstacles blocked or complicated migration:

  • Legacy technology stacks — Older frameworks and JVM versions not supported on modern cloud infrastructure. Rewrites are expensive and risky.
  • OS and library version pinning — Applications depended on exact OS patch levels and library versions that were difficult to replicate in cloud-managed environments.
  • Point-to-point IP hard-coding — Without an ESB, applications dialled each other by hard-coded IP. In the cloud, IPs change constantly, silently breaking those connections.
  • Hybrid connectivity — Apps that could not move to the cloud (regulation, technology, security) still had to be reachable from cloud-hosted apps — requiring complex VPN or direct-connect solutions.
  • Regulatory and data-residency rules — Banking, healthcare, and government sectors often mandated that data remain within national borders, limiting available cloud regions.
  • No CLI for enterprise app servers — Many enterprise-grade application servers lacked scriptable command-line interfaces, making automated deployment pipelines difficult or impossible to build.
Post-Migration Pitfalls

What Went Wrong After Moving

Many enterprises adopted "lift and shift" — moving applications with minimal changes. This transplanted on-premises habits into a metered cloud billing model, with predictably expensive results.

  • Inefficient apps on the clock — Applications that ran fine with cheap sunk-cost on-prem servers generated enormous ongoing bills when hosted in the cloud.
  • 30–40% utilisation culture — Teams used to maintaining "spare headroom" on-premises continued the same practice — at pay-per-minute cloud rates.
  • Scale-out multiplied costs — Adding more instances to compensate for inefficient code multiplied the bill without fixing the root cause.
  • Security concerns persisted — Some enterprises did not trust shared cloud infrastructure for sensitive workloads, requiring expensive additional controls.
  • Vendor lock-in is real — Proprietary managed services (databases, queues, serverless runtimes) made it technically difficult and expensive to switch providers later.
⚠ Lift-and-shift is a transitional tactic, not a strategy. It defers the real architectural work and often makes the eventual rearchitecture harder by embedding cloud-hostile patterns more deeply.
