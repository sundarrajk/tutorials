The Pre-Internet Era (1950s–1990s)
  • Software used within the enterprise — limited internal user base
  • Load was predictable; hardware planned and procured in advance
  • Hardware growth was linear — add a server as you grow
  • Releases: once every 6 months to a year — "stop the world"
  • Bare metal servers in in-house data centres
  • Monitoring: tens of servers — check CPU, RAM, Disk, Network
The Internet Era (2000s–Today)
  • Apps opened to millions of retail users worldwide
  • Load became unpredictable — events, viral spikes, Black Friday
  • Hardware demand could spike 10–20× overnight
  • Releases: daily or continuous delivery
  • Cloud: pay as you use, scale instantly, zero hardware ownership
  • Monitoring: millions of containers — metrics, logs, traces, SLOs
Era 1 · 1950s–1970s

Punch Cards & Mainframes

Computing begins — massive machines, tiny teams, no keyboards, no screens

Punch Cards
  • Programs coded by punching holes in physical cards
  • Each card = one line of code; drop the deck → start over
  • Batch submitted to operators; results next morning
  • No interactivity whatsoever
Mainframe Era
  • IBM System/360 (1964) — the landmark platform
  • Room-sized computers costing millions
  • Shared by the entire organisation via job queues
  • Still run banking & airline cores today
Key Characteristics
  • Users: a handful of operators — never end users
  • Load: completely predictable batch windows (overnight runs)
  • Data centre: purpose-built, air-conditioned vault
  • Software: custom-written assembly or early COBOL/FORTRAN

Era 2–4 · 1970s–2000s

Minis, PCs, Client-Server & the Web

Computers shrink from rooms to desks — putting power directly in users' hands — then the browser opens those desks to the entire internet

Minicomputers (1970s–80s)
  • DEC PDP, VAX — affordable for a single department, not the whole organisation
  • Multiple users via dumb terminals — keyboard + screen only, zero local CPU
  • UNIX born here — the foundation of Linux, macOS, and every cloud server today
  • First real interactive computing: type a command, get an immediate response
  • Still centralised — all compute lived on the minicomputer, terminals just displayed it
The Desktop PC Arrives (1981–90s)
  • IBM PC (1981) + Intel x86 + MS-DOS — a full computer on every desk
  • For the first time, compute power moved to the user — not just a display
  • Apple Macintosh (1984) — graphical UI, mouse; computing becomes accessible to non-engineers
  • PCs were standalone islands — each had its own files, its own applications
  • The problem this created: how do 50 PCs share data and applications?
  • Answer: connect them to a shared server → Client-Server architecture
Client-Server (mid-1980s–90s)
  • Thick client: business logic and UI run on the PC; the server holds shared data and handles heavy queries
  • LAN connects PCs to a central database server — first time a network was essential to the application
  • SAP R/3, Oracle Forms, Lotus Notes, Microsoft Exchange — all thick client-server
  • Users internal only — employees on the corporate LAN behind a firewall
  • Scaling meant buying a bigger server; no concept of serving the public internet
The Web Era (mid-1990s+)
  • Browser becomes the universal thin client — nothing to install on the PC
  • 3-tier: Browser → Web Server → App Server → Database (no thick client needed)
  • Firewall opens Port 80/443 to the internet for the first time — public users can now reach the app
  • First wave: internet banking, e-retail, online insurance, travel booking
  • Load grows — but slowly and predictably; still mostly existing customers
COTS & SOA / ESB (late 1990s–2000s)
  • Enterprises stop building everything — buy off-the-shelf: SAP (ERP), Siebel (CRM), Amdocs (Billing)
  • 10 systems × 10 peers = 90 point-to-point integrations — spaghetti
  • ESB solution: single hub, all systems plug in once — SOAP, REST, JMS routing, transformation, orchestration
  • Telecom: CRM → ESB → Prepaid / Postpaid / Network OSS / Digital Services

Inflection Point · 1998–2007

The Giants Arrive & Everything Changes

Four companies rewired what the world expects from software — and from infrastructure

🛒 Amazon — A to Z
  • Books (1994) → every product A to Z
  • Black Friday / Prime Day → load spikes 10–20× overnight
  • Must provision for peak, but 85% idle the rest of the year
  • Drove invention of elastic, on-demand infrastructure
🔍 Google — Search & Ads
  • 9/11 (2001): global search traffic spikes in minutes
  • Any world event = instant, unpredictable surge
  • Proved: advertising revenue scales with users → feature velocity = market share
  • Drove: release features fast → attract users → earn ads → fund features
📱 Facebook / Social
  • Orkut (2004), Facebook (2004) — photos, comments, likes, shares
  • User base = everyone on the planet
  • Viral content creates massive, unpredictable spikes
  • Feature wars: each new reaction required faster release cycles
🎬 Netflix — Streaming Video
  • Killed Blockbuster with streaming (2007)
  • New popular series: millions stream simultaneously
  • Peak: 15%+ of all North American internet traffic
  • Drove invention of Chaos Engineering & microservices architecture
The Pattern: All four faced the same problem — unpredictable load spikes requiring infrastructure far beyond daily average. They ended up with vast surplus capacity at off-peak times. The question became: what do you do with spare data centres?
Parallel Story · 2000–2010s

Enterprises Open Up & Salesforce Plants the Seed

Internal systems face the internet. The SaaS model shows enterprises they don't need to own anything.

Enterprises Go Online
  • Internet & Mobile Banking — 24×7 availability becomes mandatory
  • e-Commerce — festive sales create the same spike pattern as Amazon
  • Online Insurance, Just-in-Time supply chains, partner portals
  • Every enterprise opening up needs CRM — enter Salesforce
Salesforce — "No Software" (1999)
  • Old model: buy CRM licence ($500K+), buy servers, hire DBAs → 18 months to value
  • Salesforce: browser-based, pay per seat per month → live in days
  • No software to install. No servers to own. Just pay and use.
  • Enterprises got their first taste of: Software as a Service
  • CAPEX (buy upfront) → OPEX (pay monthly) — CFOs loved it
The Penny Drops: If I don't need to own the CRM software or its servers — and I just pay for what I use — why do I need to own any infrastructure? The question that seeded the cloud transition.
Birth of The Cloud · 2006

Amazon's Serendipity — The Cloud is Born

Amazon built infrastructure for peak retail load. The idle off-peak capacity became a product.

The Insight (2002–2004)
  • Amazon engineers frustrated by slow internal infrastructure provisioning
  • Standardised all internal services via APIs — same interface for everything
  • Realised: if they offered this externally, idle servers between peaks earn revenue
  • S3 — Simple Storage Service launched March 2006
  • EC2 — Elastic Compute Cloud launched August 2006
  • "Elastic" = scale up and scale back down on demand, pay by the hour
What Enterprises Gained
  • Spin up 1,000 servers in minutes (not months of procurement)
  • No data centre to build, own, or maintain
  • Pay only for what you actually use — scale to zero at night
  • Handle Black Friday; scale back Monday; don't pay for idle capacity
  • Google App Engine (2008), Microsoft Azure (2010) quickly followed
  • Competition drove prices down 80%+ over the following decade
CAPEX → OPEX: No more buying hardware upfront and depreciating it over 5 years. Cloud turned infrastructure from a capital asset on the balance sheet into a monthly utility bill — like electricity. You don't own the power plant.
Consequence · 2010s

Software Delivery Transforms

From yearly big-bang releases to continuous deployment — the cloud enabled a completely different pace

The Old Way (Pre-Cloud)
  • Release once every 6–12 months — months of preparation
  • "Stop the world" — all users impacted simultaneously
  • Change freeze, rollback plans, war-rooms during go-live
  • Big bugs wait 6 months for the fix to reach production
  • Feature backlog grows; customers frustrated; competitors catch up
The New Way (Cloud-Native)
  • Deploy multiple times per day — Amazon: thousands of deploys/day
  • Feature flags: release to 1% of users first, validate, then roll out
  • A/B testing: try two versions simultaneously, measure, pick winner
  • Bug fixed: deployed in hours, not six months
  • Continuous Integration / Continuous Delivery (CI/CD) pipelines

Summary

Five Key Ideas

If you remember nothing else — remember these

① Scale Changed Everything
  • When billions of users arrived, every assumption about infrastructure, software delivery, and operations had to be rebuilt from scratch
② Unpredictability Drove Innovation
  • The inability to predict load forced invention of elastic computing — you cannot pre-buy hardware for a viral tweet or a global news event
③ The Cloud Was Accidental
  • Amazon didn't set out to build a cloud. They solved their own problem. Monetising spare capacity was serendipity — but it changed the world
④ Salesforce Changed the Deal
  • Proving enterprises would happily rent software they previously owned planted the seed for full cloud adoption across all infrastructure
⑤ Software Velocity = Competitive Advantage
  • Companies that ship features fastest attract the most users. Users attract ad revenue. Revenue funds more features. Cloud enables this velocity. This is the flywheel that runs today's tech economy — and it's still accelerating.
